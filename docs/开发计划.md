# Slow Platform 开发计划 (V1)

根据需求文档与架构设计，将开发过程拆解为 5 个阶段。建议按照顺序逐步推进，每个阶段完成后进行联调测试。

## 阶段一：基础设施与基础资源管理
**目标**：跑通前后端基础流程，完成用户、项目、环境的基础管理。

- [ ] **1.1 数据库与基础配置**
  - [后端] 配置 SQLAlchemy 连接 MySQL。
  - [后端] 初始化 Alembic 迁移环境。
  - [后端] 定义 User, Project, Environment 基础 Model。
- [ ] **1.2 用户认证模块**
  - [后端] 实现注册、登录接口 (JWT)。
  - [前端] 实现登录页面与 Token 存储 (Store)。
  - [前端] 实现路由守卫 (Auth Guard)。
- [ ] **1.3 项目与环境管理**
  - [后端] 实现 Project CRUD 接口。
  - [后端] 实现 Environment CRUD 接口（含 Header/Config 管理）。
  - [前端] 实现项目列表、创建项目页面。
  - [前端] 实现环境管理页面。

## 阶段二：接口管理与核心执行引擎
**目标**：实现接口的录入，并能发送 HTTP 请求，跑通最核心的“发包”能力。

- [ ] **2.1 接口管理模块**
  - [后端] 定义 API 接口模型 (Api, ApiRequestTemplate)。
  - [后端] 实现接口目录树/列表与 CRUD。
  - [前端] 实现接口管理页面（树形展示、详情编辑）。
- [ ] **2.2 核心执行引擎 (HttpRunner Lite)**
  - [后端] 封装 Python `requests`，实现通用的请求发送函数。
  - [后端] 实现“调试执行”接口：接收临时请求参数 -> 发送请求 -> 返回详细响应。
  - [前端] 实现“接口调试”组件（类似 Postman 界面），支持选择环境发送请求并查看响应。

## 阶段三：用例管理与断言体系
**目标**：将接口固化为用例，并增加自动化判断逻辑。

- [ ] **3.1 单接口用例管理**
  - [后端] 定义 TestCase 模型 (含 Assertions)。
  - [后端] 实现用例 CRUD。
  - [前端] 实现用例编辑页面（支持添加断言）。
- [ ] **3.2 断言引擎**
  - [后端] 实现断言校验逻辑 (StatusCode, JSONPath, ResponseTime)。
  - [后端] 升级执行接口，支持执行用例并返回断言结果。
  - [前端] 展示断言结果（通过/失败详情）。

## 阶段四：场景编排与变量机制
**目标**：实现多接口串联，解决参数传递问题。

- [ ] **4.1 场景管理**
  - [后端] 定义 Scene, SceneStep 模型。
  - [后端] 实现场景 CRUD 与步骤编排（排序、启用/禁用）。
  - [前端] 实现场景编排画布或列表（拖拽排序）。
- [ ] **4.2 变量提取与传递 (Context)**
  - [后端] 实现变量提取器 (Extractor)：从响应中提取值存入 Context。
  - [后端] 实现模板渲染引擎 (Jinja2 或 字符串替换)：在请求中使用 `{{var}}` 替换 Context 变量。
  - [后端] 实现场景执行逻辑：按序执行 Step，维护 Session Context。

## 阶段五：计划、报告与定时任务
**目标**：批量执行与无人值守执行。

- [ ] **5.1 测试计划与报告**
  - [后端] 定义 TestPlan, TestReport, TestResult 模型。
  - [后端] 实现计划执行接口（异步 Task）。
  - [后端] 实现报告查询接口。
  - [前端] 实现计划管理与报告详情页。
- [ ] **5.2 定时任务与通知**
  - [后端] 集成 APScheduler 实现定时任务。
  - [后端] 实现 Webhook 通知逻辑。
